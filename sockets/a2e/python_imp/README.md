## 📋 Краткое описание

A2E — библиотека для организации взаимодействия между процессами по схеме **клиент-сервер** через UNIX domain sockets. Поддерживает:

- **REQUEST-RESPONSE** транзакции
- Уведомления о **прогрессе** выполнения (PROGRESS)
- Неблокирующий I/O с таймаутами
- Разбиение данных на чанки для передачи больших объёмов
- Детальное логирование

## 📖 Подробное описание работы

### Протокол взаимодействия

Одна транзакция состоит из следующих этапов:

```
┌─────────┐                              ┌─────────┐
│ Клиент  │                              │ Сервер  │
└────┬────┘                              └────┬────┘
     │                                        │
     │────────── connect() ──────────────────│
     │                                        │
     │────────── REQUEST (data) ─────────────│
     │                                        │
     │◄───────── PROGRESS (опционально) ─────│
     │                                        │
     │◄───────── RESPONSE (data) ────────────│
     │                                        │
     │────────── disconnect() ───────────────│
     │                                        │
```

### Сообщения

Каждое сообщение состоит из **заголовка** (12 байт) и **тела** (данные):

```
┌─────────────┬─────────────┬─────────────┐
│ type (4B)   │ magic (4B)  │ len (4B)    │
├─────────────┼─────────────┼─────────────┤
│ REQUEST     │ 0xBADA77BC  │ размер тела │
│ RESPONSE    │ 0xBADA77BC  │ размер тела │
│ PROGRESS    │ 0xBADA77BC  │ 0           │
└─────────────┴─────────────┴─────────────┘
```

### Машина состояний

**Сервер:**
```
NULL → IDLE → REQ_RX_START → REQ_RX → REQ_RX_FINISH
              → REQ_PROGRESS (опционально)
              → RSP_TX → RSP_TX_FINISH → IDLE
```

**Клиент:**
```
NULL → IDLE → REQ_TX_START → REQ_TX → REQ_TX_FINISH
              → REQ_PROGRESS (получение)
              → RSP_RX → RSP_RX_FINISH → IDLE
```

### Статусы операций

| Статус | Значение |
|--------|----------|
| `OK` | Успешное завершение |
| `CONTINUE` | Операция началась, нужно продолжить вызов |
| `CONTINUE_TIMEOUT` | Таймаут во время продолжения операции |
| `TIMEOUT` | Таймаут операции |
| `IN_PROGRESS` | Получено уведомление о прогрессе |
| `WRONG_STATE` | Операция недопустима в текущем состоянии |
| `INVALID_ARGS` | Неверные аргументы |
| `NO_MEM` | Ошибка выделения памяти |
| `ERROR` | Общая ошибка |

## 📚 Интеграция в клиентский код

### 1. Импорт библиотеки

```python
import sys
sys.path.insert(0, 'path/to/python_imp')

from a2e import *
```

### 2. Серверная реализация

```python
from a2e import *

def main():
    # Включить логирование
    dbg_on()
    
    # Создать конфигурацию
    cfg = cfg_set_default()
    cfg.name = "MyServer"
    cfg.sock_dir = "/tmp"
    
    # Инициализировать сервер
    server = init_server(cfg)
    
    try:
        while True:  # Цикл обработки запросов
            # Получить запрос от клиента
            status, data = request_rx(server, timeout_ms=500)
            
            if status == Status.OK:
                # Отправить уведомление о прогрессе (опционально)
                progress_tx(server, timeout_ms=500)
                
                # Обработать данные...
                response_data = process_data(data)
                
                # Отправить ответ
                response_tx(server, response_data, timeout_ms=500)
                
                # Дождаться завершения от клиента
                request_complete_wait(server, timeout_ms=500)
                
    except KeyboardInterrupt:
        pass
    finally:
        close(server)

if __name__ == "__main__":
    main()
```

### 3. Клиентская реализация

```python
from a2e import *

def main():
    dbg_on()
    
    cfg = cfg_set_default()
    cfg.name = "MyClient"
    cfg.sock_dir = "/tmp"
    
    client = init_client(cfg)
    
    try:
        # Подготовить данные запроса
        request_data = b"Hello, server!"
        
        # Отправить запрос
        status = request_tx(client, request_data, timeout_ms=500)
        
        if status == Status.OK:
            # Получить ответ (возможно с прогрессом)
            while True:
                status, response = response_rx(client, timeout_ms=500)
                
                if status == Status.IN_PROGRESS:
                    print("Server is processing...")
                    continue
                
                if status == Status.OK:
                    print(f"Response: {response}")
                    break
                
                if status == Status.TIMEOUT:
                    break
        
        # Завершить транзакцию
        request_complete(client)
        
    finally:
        close(client)

if __name__ == "__main__":
    main()
```

### 4. Конфигурация

```python
cfg = A2EConfig(
    name="MyApp",                    # Имя для логов
    sock_dir="/var/run/myapp",       # Директория для сокета
    rw_chunk_size=32 * 1024,         # Размер чанка (байт)
    req_cmplt_wait_tries=5           # Попытки ожидания завершения
)
```

### 5. Обработка ошибок

```python
from a2e import *

try:
    server = init_server(cfg)
except A2EError as e:
    print(f"Ошибка инициализации: {e.status} - {e.message}")

try:
    status, data = request_rx(server, 500)
except A2ETimeout:
    print("Превышено время ожидания")
except A2EInvalidState:
    print("Недопустимое состояние")
```

## ⚠️ Ограничения библиотеки

| Ограничение | Описание |
|-------------|----------|
| **Одно подключение** | Сервер обрабатывает только одного клиента одновременно |
| **UNIX сокеты** | Работает только на Unix-подобных системах (Linux, macOS) |
| **Один сокет** | Все клиенты подключаются к одному сокету `/tmp/a2e.sock` |
| **Нет авторизации** | Любой процесс может подключиться к сокету |
| **Нет шифрования** | Данные передаются в открытом виде |
| **Синхронный I/O** | Нет поддержки асинхронных операций (asyncio) |
| **Нет пула соединений** | Сервер не использует пул потоков/процессов |
| **Макс. размер сообщения** | Ограничен `UINT_MAX` (4 GB), но ограничен памятью |

## ⚠️ Потенциальные проблемы

### 1. Гонки при одновременном подключении

Сервер принимает только одно подключение. Если несколько клиентов подключаются одновременно, только первый будет обслужен.

**Решение:** Реализовать очередь подключений или пул потоков.

### 2. Утечка сокетов

При аварийном завершении клиента сокет может остаться в состоянии `CLOSE_WAIT`.

**Решение:** Всегда вызывать `close()` в блоке `finally`.

### 3. Блокировка при больших данных

При передаче больших объёмов данных (>100 MB) операция может занять значительное время.

**Решение:** Увеличить таймауты или разбивать данные на меньшие части.

### 4. Отсутствие проверки прав доступа

Любой пользователь может подключиться к сокету.

**Решение:** Установить права на сокет через `os.chmod()`.

### 5. Проблемы с таймаутами

Короткие таймауты могут привести к преждевременному завершению операций.

**Решение:** Подбирать таймауты экспериментально для вашей среды.

### 6. MSG_PEEK и производительность

Метод `recv(1, MSG_PEEK)` в `req_cmplt_wait()` может негативно влиять на производительность при высоких нагрузках.

**Решение:** Оптимизировать логику ожидания завершения.

### 7. Отсутствие тестов

Библиотека не имеет покрытия unit-тестами.

**Решение:** Написать тесты с использованием `pytest` или `unittest`.

### 8. Совместимость с C-версией

Протокол совместим с оригинальной C-библиотекой, но могут быть различия в обработке краевых случаев.

**Решение:** Тестировать взаимодействие между Python и C реализациями.
